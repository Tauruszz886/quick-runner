# Zulijian 场景结构备忘录

本项目从 `../zulijian` 迁移玩法时，`quick-runner` 的目标不是继续运行时生成整张地图，而是在编辑器里创建静态场景单位。所有后续代码和编辑器创建脚本都必须按“出生地”和“关卡”拆分，不能只靠单位名字区分。

## 编辑器对象树

采用场景对象树层级组织：

```text
QR_地图_ROOT
├── QR_出生地_ROOT
│   ├── QR_出生地_地板
│   ├── QR_出生地_墙...
│   └── QR_出生地_出生地砖
├── QR_第01关_ROOT
│   ├── QR_第01关_地板
│   ├── QR_第01关_墙...
│   └── QR_第01关_关卡组件...
├── QR_第02关_ROOT
...
└── QR_第10关_ROOT
```

规则：

- `module=0` 只属于 `QR_出生地_ROOT`。
- `module=1..10` 分别属于 `QR_第01关_ROOT` 到 `QR_第10关_ROOT`。
- 编辑器创建脚本必须先创建 ROOT，再创建子组件，并用 `EditorAPI.unit_add_child(parent_uid, child_uid)` 建立父子关系。
- 脚本必须幂等：同名 ROOT 或组件已存在时跳过，不重复创建。
- 出生地和关卡不能混挂到同一个父节点下。

## 本地代码目录

本地代码也按同样边界整理：

```text
ts_src/zlj/
├── birth/              # 出生点、复活点、回出生地
├── levels/             # 关卡数据；后续按 level_01..level_10 拆分
├── runtime/            # 玩家运行时、速度、机关绑定、事件注册
├── editor_scene/       # 编辑器创建计划/类型/辅助代码
├── config.ts
└── layout.ts
```

规则：

- 出生地逻辑只放 `birth/`。
- 关卡数据和关卡专属逻辑必须按关卡拆分，优先使用 `levels/level_01/`、`levels/level_02/` 这种目录。
- 编辑器场景创建逻辑放 `editor_scene/` 或 `tools/zlj_editor_scene/`，不要写进运行时入口。
- `runtime/` 只保留试玩时需要运行的逻辑，例如玩家速度、死亡回出生、对编辑器已创建单位的运行时绑定。
- quick-runner 不再通过运行时代码批量创建地板、墙、天花板和关卡地形。

## 当前迁移注意点

- `../zulijian` 的 `moduleIndex=0` 是“出生地”，`1..10` 是实际关卡。
- `../zulijian` 原先 `RUNTIME_COPY_COUNT=17` 会创建第 11 到第 17 个空模块；quick-runner 暂时只保留到第 10 关。
- 第 3/4/8/9/10 关包含运行时机关逻辑。编辑器创建静态单位后，运行时逻辑需要改成“按对象树/名字绑定已有单位”，不能依赖创建单位时返回的句柄。
- 地形、墙、地板、天花板、静态地图块必须在编辑器编辑态创建和维护；功能逻辑、机关逻辑、触发、计时、玩家交互和状态切换仍然放在运行时代码里。不要把机关运行时逻辑硬塞进编辑器创建脚本，也不要让运行时代码重新批量创建地形。
- 出生地和第 1-10 关都必须有一整块基础夹层，用来遮住大地板并给后续美术设计留中间层。命名为 `QR_出生地_基础夹层`、`QR_第01关_基础夹层` 到 `QR_第10关_基础夹层`，尺寸统一为 `160 x 0.1 x 100`，中心高度 `y=3.05`。
- 第 5 关另外有机关夹层，命名为 `QR_第05关_夹层A` 到 `QR_第05关_夹层E`。这组是玩法机关层，不能替代每关通用的基础夹层。
- quick-runner 的地图排布必须先固定边界墙，再反推出生地大地板和后续关卡。当前以 `QR_出生地_东墙 / 1205851159` 为出生地东边界定位墙，锁定其中心 `x=998.5`；出生地 footprint 为 `160 x 100`，因此出生地大地板中心为 `x=919.0`（`998.5 - (160 / 2 - 0.5)`）、`z=-36`。quick-runner 编辑器内由原生组件 `1681499584` 承接并命名为 `QR_出生地_地板`。注意 `1681499584` 是编辑器 90004 地板模型，不能把世界尺寸 `(160, 3, 100)` 直接写进它的 scale；脚本需换算成 90004 的编辑器 scale。
- 每关大地板是底层承托，必须设置在基础夹层、机关夹层和地砖/关卡地形块下面。当前约定是大地板 `y=0`，使用 `model_id=90004` 的编辑器 scale 语义；出生地和第 1-10 关 footprint 均为 `160 x 100`，关卡从出生地西边界顺接。基础夹层和第 5 关机关夹层初始下沉位 `y=3.05`，普通地砖/关卡地形块中心 `y=3.5`、高度 `3`、顶面 `y=5.0`。编辑器内置出生点（`分阵营出生点`、`蛋仔：试玩出生点`）放在出生地砖顶面 `y=5.0`；角色运行时出生高度仍由 `BIRTH_SPAWN_Y=8` 控制。
- 关卡几何源数据维护在 `data/zlj/levels/level_XX.json`；`terrain.ts` 是生成物，不再作为编辑器创建脚本的输入。
- 触发区域和玩法组件必须显式建模，不能靠运行时从 terrain 反推。当前做法是：需要运行时特殊处理的单位在场景里写入自定义 KV，例如 `QRRole`、`QRModule`、`QRComponent`、`QRMoveZ`、`QRMoving`。运行时从 `QR_地图_ROOT` 递归扫描场景单位，只处理带 `QRRole` 的单位，并从单位自身读取位置和尺寸。`data/zlj/runtime_scene_bindings.json` 暂时只是迁移输入，用于让创建脚本批量写入这些 KV；它不再被运行时加载。掉坑死亡区由 `data/zlj/fall_death_zones.json` 维护，并在编辑器场景中创建带 `QRRole="fall_death"` 的触发器组件。日志标签为 `ZLJ_HOLE_DEATH`，不要再叫“空洞返回”。

## FBX / SVG 标注工作流

后续用户给 FBX 或从 FBX 拆出地图组件后，除了在编辑器里创建单位和整理对象树，还必须同步维护可视化索引：

- 组件落到编辑器后，先按关卡导出“SVG编号 / 数据结构名 / 编辑器组件名 / unit_id / 坐标 / 尺寸”索引表。第 3 关当前索引见 `docs/level_03_component_index.md`，生成脚本见 `tools/zlj_editor_scene/export_level_component_index.py`。
- 如果该关已有 SVG 图，优先修改之前引用的 SVG 文件，不重新创建替代图。
- SVG 图上必须可见标出关键组件的编号、编辑器 `unit_id` 和完整组件名，例如 `11 / ID 1032584478 / QR_第03关_dxf_84C_24x17_1875`。
- 当用户在编辑器里改了组件尺寸、位置或形状，本地关卡数据、编辑器创建计划、索引表、SVG 图必须一起更新，保证能从图上的编号反查到代码数据结构和编辑器实例。
- SVG 标注按关卡和功能组织，不把不同关卡组件混到同一张无索引图里。

## DWG / CAD 修改工作流

当用户要求按 SVG 或编辑器结果修改 DWG/CAD 文件时，按同一条编号链路处理：

- 先用 ODA File Converter 把 DWG 转为 DXF，再用 `ezdxf` 读取，不直接猜 DWG 二进制结构。
- CAD 实体 handle、代码数据结构编号、SVG 编号和编辑器实例必须一起记录。例如第 3 关当前关键项：SVG编号 `11` / CAD handle `84C` / `dxf_84C_24x17_1875` / `QR_第03关_dxf_84C_24x17_1875` / unit_id `1032584478`。
- 如果 CAD 内有 `3DSOLID`，必须优先修改实体本身的 ACIS/SAB 数据；2D 线框或投影线只能作为辅助同步，不能只改线框。
- 修改后输出 DXF，并在可行时再用 ODA 转回 DWG；最终要把新 DWG 反转 DXF 读回校验目标实体尺寸。
- 第 3 关 `84C` 的当前 CAD 结果是把原局部矩形 `x=92..116, z=29.6875..46.875` 改为 `x=94..114, z=29.6875..46.875`，中心保持不变。此前的一次性修改脚本已删除，不作为长期工具维护。
